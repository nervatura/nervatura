1. Installation

  • Nervatura packages

    Nervatura Core Package:

    $ npm install nervatura --production --save

    Package Framework Admin NDI NPI Report Docs Comment
    nervatura-azure Azure Functions SLS Admin yes yes yes online
    nervatura-express Express Framework NAS Admin yes yes yes contain Additional example programs and downloads
    nervatura-gcloud Google Cloud Functions SLS Admin yes yes yes online Experimental feature!
    nervatura-lambda AWS Lambda SLS Admin yes yes yes online Experimental feature!
    nervatura-openwhisk IBM Cloud Functions SLS Admin yes yes yes online Experimental feature!

  • Quick Start
    • Nervatura with Express Framework

      $ git clone https://github.com/nervatura/nervatura-express.git nervatura
      $ cd nervatura
      $ npm install --production

    • development mode

      $ npm run dev

      and http://localhost:8080/

    • production mode

      $ npm start

      and https://localhost:8080/

2. Cloud Hosting Services

  • Amazon Web Services

    Elastic Beanstalk

    If you're not already an AWS customer, you'll need to sign up. Signing up allows you to access Elastic Beanstalk and other AWS services that you will need. The Elastic Beanstalk Command Line Interface (EB CLI) is a command line client that you can use to create, configure, and manage Elastic Beanstalk environments. Download and install the last Amazon EB CLI.

    Clone the latest Nervatura Express version:

    $ git clone https://github.com/nervatura/nervatura-express.git nervatura_aws

    Run eb init in your application's project directory to configure the EB CLI and your project:

    $ cd nervatura_aws

    $ git init

    $ eb init [APP_NAME] --platform node.js

    Add new and changed files and create a new environment:

    $ git add .

    $ git commit -a -m 'latest'

    $ eb create [ENVIRONMENT_NAME] --cname [SUBDOMAIN_NAME] --platform node.js --elb-type application

    Launch your browser and view the app, by running the following command:

    $ eb open

    Lambda Functions*

    Download and install the last Amazon AWS CLI. Clone the latest Nervatura Lambda version:

    $ git clone https://github.com/nervatura/nervatura-lambda.git

    $ cd nervatura-lambda

    Create an Amazon S3 bucket:

    $ aws s3 mb s3://[BUCKET_NAME] --region [REGION]

    Package Nervatura template:

    $ aws cloudformation package --template-file template.yaml \
      --output-template-file ntura.yaml --s3-bucket [BUCKET_NAME]

    Deploy packaged Nervatura template:

    $ aws cloudformation deploy --template-file ./ntura.yaml \
      --stack-name [STACK_NAME] --capabilities CAPABILITY_IAM

    * Experimental feature! Use only for testing!

    Amazon (OAuth 2.0) authentication (Express Framework)

    The Amazon authentication strategy authenticates users using an Amazon account and OAuth 2.0 tokens. The client ID and secret are obtained by registering an application at the Login with Amazon Developer Center.

    Before you deploy the application, modify the following section in the node-settings.config file:

    AMAZON_CLIENT_ID: YOUR_AMAZON_CLIENT_ID
    AMAZON_CLIENT_SECRET: YOUR_AMAZON_CLIENT_SECRET

    1. Login to NAS Admin GUI

    • Username: admin
    • Password: leave the field empty, not needed (Default. Please change after the first login!)

    2. Add a new user (User Accounts)

    • Username: a valid(!) Amazon user
    • Password: leave the field empty, not needed

    3. Remove the admin user (optional) and modify the following setting (Server Settings):

    • Fieldname: nas_auth
    • Value: amazon

    4. Log off and re-enter the new user.

    Amazon RDS DB Instance

    You can use an Amazon Relational Database Service (Amazon RDS) DB instance to store data gathered and modified by your application. The database can be attached to your environment and managed by Elastic Beanstalk, or created and managed externally.

    Adding a DB Instance to Your Environment:

    • Open the Elastic Beanstalk console.
    • Navigate to the management page for your environment.
    • Choose Configuration.
    • Under Data Tier, choose Create a new RDS database.
    • Choose a DB engine, enter a user name and password, and then choose Apply.

    Or create a new environment with a database:

    $ eb create [ENVIRONMENT_NAME] --cname [SUBDOMAIN_NAME] --platform node.js --elb-type application \
      --database --database.engine [DB_ENGINE] --database.username [DB_USER] --database.password [DB_PASSWORD]

    For production environments, create a DB instance outside of your Elastic Beanstalk environment to decouple your environment resources from your database resources. Using an external DB instance lets you connect to the same database from multiple environments and perform blue-green deployments. For instructions, see Using Elastic Beanstalk with Amazon Relational Database Service.

    PostgeSQL instance:

    Enable the Cloud SQL API. Creating an instance and setting the password:

    $ gcloud sql instances create [INSTANCE_NAME] --database-version=POSTGRES_9_6 \
    --cpu=[NUMBER_CPUS] --memory=[MEMORY_SIZE]

    $ gcloud sql users set-password postgres no-host --instance=[INSTANCE_NAME] \
    --password=[PASSWORD]

    See more here: Creating PostgreSQL Instances

    MySQL instance:

    Enable the Cloud SQL API. Creating an instance and setting the password:

    $ gcloud sql instances create [INSTANCE_NAME] --tier=[MACHINE_TYPE] --region=[REGION]

    $ gcloud sql users set-password root % --instance [INSTANCE_NAME] --password [PASSWORD]

    See more here: Creating MySQL Instances

    To allow your app to connect to your Cloud SQL instance when the app is deployed, add the user, password, database, and instance connection name variables from Cloud SQL to the related environment variables in the app.yaml file: The deployed application will connect via unix sockets.

    env_variables:
      SQL_USER: YOUR_SQL_USER
      SQL_PASSWORD: YOUR_SQL_PASSWORD
      SQL_DATABASE: YOUR_SQL_DATABASE
    # e.g. my-awesome-project:us-central1:my-cloud-sql-instance
      INSTANCE_CONNECTION_NAME: YOUR_INSTANCE_CONNECTION_NAME

    beta_settings:
      cloud_sql_instances: YOUR_INSTANCE_CONNECTION_NAME

    See more here: Using Cloud SQL for MySQL and Using Cloud SQL for PostgreSQL

  • Microsoft Azure

    If you don't have an Azure subscription, create a free account before you begin. Install or upgrade the last version of the Azure CLI 2.0.

    Azure Web Apps

    Create a web app from a local Git repository.

    • Create a resource group (see more list-locations):

      $ az group create --location northeurope --name [GROUP_NAME]

    • Create an App Service plan (in FREE tier):

      $ az appservice plan create --name [PLAN_NAME] --resource-group [GROUP_NAME] --sku FREE

      Linux and Basic tier sample:

      $ az appservice plan create --name [PLAN_NAME] --resource-group [GROUP_NAME] \
        --is-linux --number-of-workers 3 --sku B1

    • Create a web app (see more list-runtimes):

      $ az webapp create --name [APP_NAME] --resource-group [GROUP_NAME] --plan [PLAN_NAME] \
        --deployment-local-git --runtime "node|8.1"

    • Set the account-level deployment credentials:

      $ az webapp deployment user set --user-name [USERNAME] --password [PASSWORD]

    • Set some environment variables:

      $ az webapp config appsettings set --resource-group [GROUP_NAME] --name [APP_NAME] \
        --settings HOST_TYPE=azure

    • Set Node.js development configurations (optional, if there is no SSL support):

      $ az webapp config appsettings set --resource-group [GROUP_NAME] --name [APP_NAME] \
        --settings NODE_ENV=development

    • Clone and configure the (empty) local git directory:

      $ git clone https://[USERNAME]@[APP_NAME].scm.azurewebsites.net/[APP_NAME].git

      And upgrade to the latest Nervatura Express version (Linux sample):

      $ git clone https://github.com/nervatura/nervatura-express.git nervatura_latest
      $ rm -r [APP_NAME]/*
      $ cp -r nervatura_latest/* [APP_NAME]/
      $ cd [APP_NAME]
      $ git add .
      $ git commit -a -m 'latest'
      $ git push

    • Or all in one step (if you don't want to modify anything):

      $ az webapp deployment source config --resource-group [GROUP_NAME] --name [APP_NAME] \
        --repo-url https://github.com/nervatura/nervatura-express.git --branch master --manual-integration

    See the web app here: http://app_name.azurewebsites.net

    MySQL in-app feature:

    See more Announcing MySQL in-app for Web Apps (Windows)

    MySQL server:

    Create an Azure MySQL server.

    $ az mysql server create --resource-group [GROUP_NAME] --name [SERVER_NAME] \
      --admin-user [ADMIN_NAME] --admin-password [ADMIN_PASSWORD] \
      --performance-tier Basic --compute-units 50 --ssl-enforcement Disabled

    Configure a server-level firewall rule:

    $ az mysql server firewall-rule create --resource-group [GROUP_NAME] --server [SERVER_NAME] \
      --name AllowYourIP --start-ip-address 0.0.0.0 --end-ip-address 255.255.255.255

    Create an empty database for MySQL server. Connect to your server using the mysql command-line tool:

    $ mysql -h [SERVER_NAME].mysql.database.azure.com -u [ADMIN_NAME]@[SERVER_NAME] -p

    mysql> CREATE DATABASE nervatura;

    Set some environment variables:

    $ az [webapp/functionapp] config appsettings set --resource-group [GROUP_NAME] --name [APP_NAME] \
      --settings AZURE_MYSQL_HOST=[SERVER_NAME].mysql.database.azure.com

    $ az [webapp/functionapp] config appsettings set --resource-group [GROUP_NAME] --name [APP_NAME] \
      --settings AZURE_MYSQL_USERNAME=[ADMIN_NAME]@[SERVER_NAME]

    $ az [webapp/functionapp] config appsettings set --resource-group [GROUP_NAME] --name [APP_NAME] \
      --settings AZURE_MYSQL_PASSWORD=[ADMIN_PASSWORD]

    Enable ssl enforcement when connect to server. (Optional.)

    $ az mysql server update --resource-group [GROUP_NAME] --name [SERVER_NAME] \
      --ssl-enforcement Enabled

    See more here: Create an Azure Database for MySQL server using Azure CLI

    PostgeSQL server:

    Create an Azure PostgreSQL server. By default, postgres database gets created under your server.

    $ az postgres server create --resource-group [GROUP_NAME] --name [SERVER_NAME] \
      --admin-user [ADMIN_NAME] --admin-password [ADMIN_PASSWORD] \
      --performance-tier Basic --version 9.6 --ssl-enforcement Disabled

    Configure a server-level firewall rule:

    $ az postgres server firewall-rule create --resource-group [GROUP_NAME] --server [SERVER_NAME] \
      --name AllowAllIps --start-ip-address 0.0.0.0 --end-ip-address 255.255.255.255

    Set some environment variables:

    $ az [webapp/functionapp] config appsettings set --resource-group [GROUP_NAME] --name [APP_NAME] \
      --settings AZURE_PG_HOST=[SERVER_NAME].postgres.database.azure.com

    $ az [webapp/functionapp] config appsettings set --resource-group [GROUP_NAME] --name [APP_NAME] \
      --settings AZURE_PG_USERNAME=[ADMIN_NAME]@[SERVER_NAME]

    $ az [webapp/functionapp] config appsettings set --resource-group [GROUP_NAME] --name [APP_NAME] \
      --settings AZURE_PG_PASSWORD=[ADMIN_PASSWORD]

    Enable ssl enforcement when connect to server. (Optional. If you're getting error after receiving response from database: ECONNRESET with unpooled connection)

    $ az postgres server update --resource-group [GROUP_NAME] --name [SERVER_NAME] \
      --ssl-enforcement Enabled

    See more here: Create an Azure Database for PostgreSQL using the Azure CLI

Express Framework

  • APP_LANG: String. Default value: 'en'. Valid values: en.
  • START_PAGE: String
    • ntura: Default.
    • static: Custom static homepage
    • custom: Custom NodeJS/Express homepage
  • DATA_DIR: Path string. Default: app_home/data. A persistent data directory. Subfolders: database(sqlite), data(export/import), storage.
  • REPORT_DIR: Path string. Default: nervatura_module/report. The Nervatura Report templates directory.
  • PYTHON2_PATH: Path string. Default:'python'. The path where to locate the "python" executable. Optional, for the server-side Nervatura Report.
  • DATA_STORE: String.
    • memory
    • lokijs: Default
    • pouchdb
    • datastore: Google Datastore (Google App Engine default)
  • LONG_TIMEOUT: Integer. Default: (4 * 60 * 1000). Express response timeout/msec. (create/export/import database and import demo database)
  • SESSION_SECRET: String. Default: auto. See more cookie-session
  • COOKIE_MAX_AGE: Integer. Default value: 7*24*60*60*1000. See more cookie-session
  • CONTENT_LENGTH: Integer. Default value: 20000. See more express-content-length-validator
  • ALL_HOST_RESTRICTION, NAS_HOST_RESTRICTION, NPI_HOST_RESTRICTION, NDI_HOST_RESTRICTION: String. Limiting server connections(ip pattern)
  • NAS_AUTH: String. Default:'local'. NAS Admin authentication strategy. Valid values: local (username and password), amazon (Amazon OAuth 2.0), azure (Azure AD OpenID Connect), google (Google OAuth 2.0)
  • AMAZON_CLIENT_ID, AMAZON_CLIENT_SECRET: String. NAS Admin authentication Amazon OAuth 2.0 values.
  • AZURE_CLIENT_ID, AZURE_CLIENT_SECRET, AZURE_REDIRECT_URL, AZURE_IDENTITY_URL: String. NAS Admin authentication Azure AD OpenID Connect values.
  • GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET: String. NAS Admin authentication Google OAuth 2.0 values.
  • SMTP_HOST, SMTP_PORT, SMTP_SECURE, SMTP_USER, SMTP_PASSWORD: String. nodemailer smtp options
  • MJ_APIKEY_PUBLIC, MJ_APIKEY_PRIVATE: String. mailjet: clientID, clientSecret
