1. Overview

The Nervatura Data Interface (NDI) is made for those who want to query or transmit data to a Nervatura customer database. The purpose of functions is to give an easily operable, clear and safe tool to manage while the real data storage layer is covered:

  • Easy to learn: the structure of functions follows the logic of business, it is not necessary to get acquainted with and to understand the data storing principles or the exact structure of the database
  • Safety: using it for data setting and updating, data integrity problems can be avoided, information queried this way is surely gathered
  • Easy to maintain: update of the functions for potential database changes and retroactive compatibility is centrally ensured
  • Platform and place independent use: the functions can be called by simple http requests from anywhere, from any device, from any programming language

The set of functions of NDI supports the following operations:

  • Resources
    Recording new data, data modifications and query: customer, product, employee, tool, project.
  • Documents
    Recording new data, data modifications and query: offer, order, invoice, worksheet, rental.
  • Payments and Stock control
    Bank and petty cash trade, stock movements, tool movements, production.
  • Events and additional data
    Definition, modification, deletion and query of typical additional data and object events.
  • Settings
    Company data, database settings, transaction numbering, stock and location, foreign currency, groups etc.

2. The Structure and Function Types

Functions can be called by http/https GET/POST request.

  • For recording and modifying data: updateData
    Optional parameters (params): insert_row, insert_field. In the data parameter block the object (customer, product, tools, etc.) features can be amended or new features added. These adjusments are separated with | signal (GET request) or valid JSON object (POST request).
  • Deleting data: deleteData,
  • Query of various data formats: getData

All three functions offer the possibility to enter further data and refinements in three parameters in which information is stored with "|" sign separator in the character strings.
code: empty or base64. At NAS options also encryption settings can be added.
params: it contains identification information just as the processing type and method.
data,filters: the data itself (data), possible filtration, formatting, return format information (filters) is located here according to the given function.

In all cases the functions have a return value which after successful data processing with a prefix "OK|" is the identifier of processed objects or in data query function is the defined return format. In case of handled error the message comes after "Error|" prefix. The current database performs the verification of the data type, the syntax of filter terms or given sql queries, in case of error this message is transferred.

Valid fieldname and type values in params

Name Type Comment
database* text Required, the name of the database
username* text Required (remote access must be enabled!)
password* text Required, optional MD5 encoded
datatype* text Required. See valid datatype values
use_deleted no value Optional. Show (getData) or use (updateData) the deleted data. Used for data export/import.
insert_row no value Optional, only in case of request for updateData. If used and no unique identifier corresponding to the object is set while updating the data, then the function will create it automatically in case it has not existed yet. Also new object will be created even if the given unique identifier did not exist before. Without adding this parameter both mentioned cases would result in error messages.
insert_field no value Optional, only in case of request for updateData. If used, and the field name given at data update has not existed before, it will automatically be created and the value assigned to it. The field type will always be set to text. The description of the field is identical to the name of the field. If other field type is needed than text it should be defined beforehand in deffield object. Without adding this parameter the mentioned case would result in error message.

Valid filters parameter values (getData)

Name Type Comment
output* text Required. Valid values: json, html, xml, csv. If not set, the default value: json.
header text Optional, only if the value output html. Replaces the table column headings. The new values should be given separated by a comma.
columns text Optional. Only the specified columns, in a particular order will be shown. Results in error if the column name is invalid. The new values should be given separated by a comma.
where text Optional. A valid SQL WHERE condition. Results in error if the column name is invalid or wrong type syntax.
orderby text Optional. Results in error if the column name is invalid. The new values should be given separated by a comma.
show_id no value Optional. If specified, the primary key will be shown.
cross_table no value Optional, only if the value output html.
no_deffield no value Optional. Does not show the additional data.

3. The NAS settings

The use of NDI functions can be prohibited or permitted through the surface of Nervatura Application Server (NAS) in dedicated customer database or we can give further limitations as well:

  • NDI permission or prohibition on the given customer database
  • The given password value in the "params" parameter must be MD5 coding or simple text
  • Data must be encrypted and accompanied with master password
  • IP addresses used for the connection can be limited: whole address or address range can also be given

Regardless to the setting of NAS in certain customer databases user data access is also limited. NDI takes into consideration the given options for group access, also full access of a user and/or group can be prohibited.

5. Examples

1. Enter two new customers in one request: customer name, VAT number and eye colour. If the eye colour customer feature does not exist, it will be automatically generated. We give a customer number to the first but we do not enter it for the second customer. The program will generate a new one automatically for the second customer and it will be visible in the response given to the action. If the customer number that we have given for the first customer has already existed then the data of it will be updated with the new entry.

GET REQUEST:
/ndi/updateData?code=&params=database=demo|username=demo|password=|datatype=customer|insert_row|insert_field& data=custnumber=DOC/00001|custname=NDIDoc Sample 1.|taxnumber=12345678-1-12|eye colour=blue ||custname=NDIDoc Sample 2.|taxnumber=87654321-1-21|eye colour=brown

POST DATA:
data: {"id":2,"method":"updateData","jsonrpc":"2.0","params":[ {"database":"demo","username":"demo","password":"","datatype":"customer","insert_row":true,"insert_field":true}, [{"custnumber":"DOC/00001","custname":"NDIDoc Sample 1.","taxnumber":"12345678-1-12","eye colour":"blue"}, {"custname":"NDIDoc Sample 2.","taxnumber":"87654321-1-21","eye colour":"brown"}]]}

2. A new customer feature generated by type. We generate the customer weight data that can only be a number.

GET REQUEST:
/ndi/updateData?code=&params=database=demo|username=demo|password=|datatype=deffield|insert_row|insert_field& data=fieldname=customer_weight|nervatype=customer|fieldtype=float|description=Weight

POST DATA:
data: {"id":2,"method":"updateData","jsonrpc":"2.0","params":[ {"database":"demo","username":"demo","password":"","datatype":"deffield","insert_row":true}, [{"fieldname":"customer_weight","nervatype":"customer","fieldtype":"float","description":"Weight "}]]}

3. For the customer generated in the point 1. we can set the weight by customer number.

GET REQUEST:
/ndi/updateData?code=&params=database=demo|username=demo|password=|datatype=customer|insert_row|insert_field& data=custnumber=DOC/00001|customer_weight=92
or
/ndi/updateData?code=&params=database=demo|username=demo|password=|datatype=fieldvalue|insert_row& data=fieldname=customer_weight|refnumber=DOC/00001|value=92

POST DATA:
data: {"id":2,"method":"updateData","jsonrpc":"2.0","params":[ {"database":"demo","username":"demo","password":"","datatype":"customer","insert_row":true}, [{"custnumber":"DOC/00001","customer_weight":"92"}]]}
or
data: {"id":2,"method":"updateData","jsonrpc":"2.0","params":[ {"database":"demo","username":"demo","password":"","datatype":"fieldvalue","insert_row":true}, [{"fieldname":"customer_weight","refnumber":"DOC/00001","value":"92 "}]]}

4. For the customer generated in the point 1. we add new address data.

GET REQUEST:
/ndi/updateData?code=&params=database=demo|username=demo|password=|datatype=address|insert_row|insert_field& data=nervatype=customer|refnumber=DOC/00001|rownumber=1|country=Anywhere|zipcode=ZIP1234|city=BigCity|street=BlueStreet 3.

POST DATA:
data: {"id":2,"method":"updateData","jsonrpc":"2.0","params":[ {"database":"demo","username":"demo","password":"","datatype":"address","insert_row":true}, [{"nervatype":"customer","refnumber":"DOC/00001","rownumber":"1","country":"Anywhere","zipcode":"ZIP1234","city":"BigCity","street":"BlueStreet 3. "}]]}

5. To the customer generated in the point 1. we attach a new event.

GET REQUEST:
/ndi/updateData?code=&params=database=demo|username=demo|password=|datatype=event|insert_row|insert_field& data=nervatype=customer|refnumber=DOC/00001|eventgroup=meeting|fromdate=2013-01-13 13:00:00|todate=2013-01-13 15:00:00|subject=Customer meeting

POST DATA:
data: {"id":2,"method":"updateData","jsonrpc":"2.0","params":[ {"database":"demo","username":"demo","password":"","datatype":"event","insert_row":true}, [{"nervatype":"customer","refnumber":"DOC/00001","eventgroup":"meeting","fromdate":"2013-01-13 13:00","todate":"2013-01-13 15:00","subject":"Customer meeting"}]]}

6. Querying data for a client

GET REQUEST:
/ndi/getData?code=&params=database=demo|username=demo|password=|datatype=customer&filters=output=html|where=custnumber='DOC/00001'|cross_tab

POST DATA:
data: {"id":1,"method":"getData","jsonrpc":"2.0","params":[ {"database":"demo","username":"demo","password":"","datatype":"customer"}, {"output":"html","where":"custnumber='DOC/00001'","cross_tab":true}]}

7. We delete the customer generated in the point 1.

GET REQUEST:
/ndi/deleteData?code=&params=database=demo|username=demo|password=|datatype=customer&data=custnumber=DOC/00001

POST DATA:
data: {"id":3,"method":"deleteData","jsonrpc":"2.0","params":[ {"database":"demo","username":"demo","password":"","datatype":"customer"},[{"custnumber":"DOC/00001"}]]}
